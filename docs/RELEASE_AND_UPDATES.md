# Warpforge desktop releases and updates

Status: implementation plan, based on the repository state on 2026-07-14.

This document deliberately describes the work before changing release-critical
configuration. The current desktop build is a development shell, not yet a
shippable or self-updating application.

## Recommended release model

Use GitHub Releases as both the public download page and the stable update
feed. Publish one unified Warpforge version and one release containing:

- signed desktop installers and Tauri updater artifacts;
- `latest.json` generated by `tauri-apps/tauri-action`;
- the `warpforge` daemon/CLI archives currently consumed by `install.sh`;
- checksums and release notes.

Do not create a desktop-only GitHub release while `install.sh` uses
`/releases/latest`: it would select that release and then fail if the expected
`warpforge-<target>.tar.gz` asset is absent.

For the first public launch, use a manually triggered draft release. Build all
platforms into the draft, test the downloaded installers on clean machines,
then publish it. Automatic updates should only see published, non-prerelease
stable releases.

## Current repository audit

### App structure and versions

- The native app is under `desktop/`: React 18 + Vite frontend and a Tauri 2
  shell in `desktop/src-tauri`.
- The Rust lockfile currently resolves `tauri` 2.11.5, `tauri-build` 2.6.3,
  `tauri-runtime` 2.11.3, and `tauri-utils` 2.9.3. The manifests use broad
  `"2"` / `^2.1.0` constraints, so lockfiles must be committed and CI must use
  frozen installs.
- Three desktop version declarations currently agree on `0.1.0`:
  `desktop/src-tauri/tauri.conf.json`, `desktop/src-tauri/Cargo.toml`, and
  `desktop/package.json`. The root daemon/CLI is still `0.0.3` in
  `Cargo.toml`.
- The Tauri crate is intentionally excluded from the root Cargo workspace.
  CI must therefore test/build both manifests explicitly.

### Packaging and update gaps

- `desktop/src-tauri/tauri.conf.json` has `bundle.active: false`. It cannot
  produce `.dmg`, `.app`, `.msi`, NSIS, AppImage, `.deb`, or updater bundles.
- `tauri-plugin-updater` and `tauri-plugin-process` are not installed or
  initialized. There is no update UI or check/install/relaunch flow.
- No updater public key or endpoint is configured and no updater signing key
  is supplied in CI.
- The release workflow only cross-compiles root CLI archives for macOS arm64
  and Linux arm64. It does not build/test the frontend or Tauri app, sign
  platform binaries, or create `latest.json`.
- `beforeBuildCommand` builds the root daemon, but it is not listed in
  `bundle.externalBin` or `bundle.resources`. Production code expects a
  `warpforge` sibling next to the GUI executable. That happens to resemble the
  macOS bundle layout but is not a cross-platform packaging contract.
- The current CSP permits local websocket connections, which the UI needs. It
  does not need to permit the updater endpoint because update downloading is
  performed by native Rust/plugin code.

The bundled daemon is the first release blocker: an installer that launches
but cannot find its matching daemon is not viable, and updating only the GUI
could create a protocol mismatch.

## Target platforms and artifacts

Recommended public matrix after platform smoke tests:

| Platform | Download artifact | Auto-update artifact | Priority |
| --- | --- | --- | --- |
| macOS Apple Silicon | signed/notarized `.dmg` | `.app.tar.gz` + `.sig` | launch blocker |
| macOS Intel | signed/notarized `.dmg` | `.app.tar.gz` + `.sig` | launch if tested |
| Windows x64 | signed NSIS `-setup.exe` | same installer + `.sig` | next public target |
| Linux x64 | AppImage, optionally `.deb` | AppImage + `.sig` | next public target |
| Linux arm64 | AppImage/`.deb` | AppImage + `.sig` | later/native ARM runner |

The repository has previously declared arm64-only support, so do not claim
Intel, Windows, or Linux x64 in marketing until their installed applications
can start the daemon and launch an agent successfully. Prefer NSIS over MSI in
`latest.json` (`updaterJsonPreferNsis: true`); its per-user installation is a
smoother default. `.deb` is useful for manual installation but AppImage is the
Linux artifact supported by the standard updater flow.

For a fast Product Hunt launch, macOS Apple Silicon can be the minimum honest
platform. Do not delay updater architecture for other platforms: use the same
workflow and signing model from the first release so later platforms can be
added without changing update trust.

## Tauri updater implementation

Tauri 2's updater uses two distinct signatures:

1. The Tauri updater signature proves that the downloaded update came from
   Warpforge. It is mandatory on every updater platform and uses the
   `TAURI_SIGNING_PRIVATE_KEY` secret plus a public key embedded in the app.
2. OS code signing establishes publisher identity and prevents platform
   warnings. It is separate from the updater signature. macOS signing and
   notarization are required for a polished direct download; Windows code
   signing avoids SmartScreen's unknown-publisher warning.

Official references:

- [Tauri updater plugin](https://v2.tauri.app/plugin/updater/)
- [Tauri GitHub pipeline](https://v2.tauri.app/distribute/pipelines/github/)
- [macOS signing and notarization](https://v2.tauri.app/distribute/sign/macos/)
- [Windows code signing](https://v2.tauri.app/distribute/sign/windows/)
- [`tauri-action` inputs](https://github.com/tauri-apps/tauri-action)

### Generate and store the updater key

Run once on a trusted machine (use the project's pinned Tauri CLI):

```bash
cd desktop
npm ci
npm run tauri signer generate -- -w ~/.tauri/warpforge.key
```

Store the complete private-key text in the GitHub environment secret
`TAURI_SIGNING_PRIVATE_KEY` and its password, if set, in
`TAURI_SIGNING_PRIVATE_KEY_PASSWORD`. Put only the generated public key text in
`desktop/src-tauri/tauri.conf.json`. Back up the private key in an organization
password manager/offline vault. Losing it means installed clients cannot trust
future updates; leaking it requires an explicit key-rotation release.

### Required code changes

`desktop/src-tauri/Cargo.toml`:

```toml
[dependencies]
tauri-plugin-updater = "2"
tauri-plugin-process = "2"
```

`desktop/package.json`:

```json
{
  "dependencies": {
    "@tauri-apps/plugin-process": "^2",
    "@tauri-apps/plugin-updater": "^2"
  }
}
```

Initialize both plugins in `desktop/src-tauri/src/main.rs` before `setup`:

```rust
.plugin(tauri_plugin_updater::Builder::new().build())
.plugin(tauri_plugin_process::init())
```

Add a narrow capability file such as
`desktop/src-tauri/capabilities/main.json`, scoped to window `main`, with only
`core:default`, `updater:default`, and `process:allow-restart` (use the exact
permission names generated by the installed plugin versions). Do not grant
shell or filesystem permissions merely for updating.

Change `desktop/src-tauri/tauri.conf.json` along these lines:

```json
{
  "bundle": {
    "active": true,
    "createUpdaterArtifacts": true,
    "targets": "all",
    "externalBin": ["binaries/warpforge"],
    "macOS": {
      "minimumSystemVersion": "11.0"
    }
  },
  "plugins": {
    "updater": {
      "pubkey": "<CONTENTS OF THE GENERATED PUBLIC KEY>",
      "endpoints": [
        "https://github.com/ephor/warpforge/releases/latest/download/latest.json"
      ],
      "windows": {
        "installMode": "passive"
      }
    }
  }
}
```

Do not copy that `externalBin` entry without implementing the sidecar staging
step. Tauri expects target-qualified files such as
`desktop/src-tauri/binaries/warpforge-aarch64-apple-darwin` and a Windows
`.exe` variant. Add `scripts/stage-sidecar.*` or a small Rust/Node script that:

1. derives `TAURI_ENV_TARGET_TRIPLE` (or accepts `--target`);
2. builds root `Cargo.toml` for that exact target;
3. copies the result to Tauri's target-qualified sidecar filename;
4. fails if the daemon target and app target differ.

Then replace the current `beforeBuildCommand` with that deterministic script
plus `npm run build`. Update daemon startup to resolve the packaged sidecar by
Tauri's supported path/API rather than relying solely on a sibling heuristic.
Keep `WARPFORGE_DAEMON_BIN` for development and diagnostics.

### Update UX

Add an app-level update service, for example `desktop/src/lib/updater.ts`, and
a small settings/about UI:

- check once after the UI and daemon are healthy, with a short delay;
- also expose a manual **Check for updates** action;
- show version, release notes, download progress, and explicit restart action;
- never interrupt an active task automatically;
- before install/restart, use the existing runtime/session checks and explain
  what remains running;
- debounce failures and show a retryable, non-blocking message;
- in browser/Vite mode, return a typed `unsupported` result rather than trying
  to import native plugins.

The updater must update the GUI and bundled daemon atomically in the same app
artifact. Record the daemon/protocol version in diagnostics so mismatches are
observable.

## CI/CD design

### 1. Pull-request CI (`.github/workflows/ci.yml`)

Create a separate read-only validation workflow for pull requests and `main`:

- root: `cargo fmt --check`, `cargo clippy --all-targets -- -D warnings`, and
  `cargo test --locked`;
- desktop web: `npm ci`, `npm run lint`, `npm run typecheck`, `npm test`;
- desktop Rust on at least one Linux runner after installing WebKitGTK 4.1
  dependencies: `cargo check --locked --manifest-path desktop/src-tauri/Cargo.toml`;
- optionally run an unsigned `npm run tauri build -- --no-bundle` smoke build
  on each supported OS before enabling that OS in releases.

Pin major action versions and enable dependency caching. Use `npm ci`, not a
mix of Bun and npm, until one canonical frontend package manager/lockfile is
chosen. The current repository has both `bun.lock` and `package-lock.json`.

### 2. Release workflow (`.github/workflows/release.yml`)

Replace/extend the current CLI-only workflow. Recommended trigger:

```yaml
on:
  workflow_dispatch:
    inputs:
      version:
        description: Existing vX.Y.Z tag to release
        required: true
```

Alternatively retain `push.tags: ['v*']`, but protect the release environment
with required approval. The pipeline should:

1. **Validate**: ensure the tag is exactly `v<SemVer>` and equals all four
   manifests (`Cargo.toml`, Tauri config, desktop Cargo manifest, package.json),
   ensure `CHANGELOG.md` contains the version, and run the full CI suite.
2. **Create/find one draft release** for the exact tag.
3. **Build desktop matrix** with native runners and
   `tauri-apps/tauri-action` using `projectPath: desktop`, the exact tag, draft
   release, `uploadUpdaterJson: true`, `uploadUpdaterSignatures: true`, and
   `updaterJsonPreferNsis: true`.
4. **Build CLI/daemon archives** and attach them to the same release so
   `install.sh` remains valid. Expand the CLI target matrix only where tested.
5. **Verify** the release has every expected installer, updater `.sig`, CLI
   archive, and a valid `latest.json` containing every shipped updater target.
   Download every URL from `latest.json`, verify HTTP success, and verify that
   its version equals the tag.
6. **Generate SHA-256 checksums/SBOM** (recommended: CycloneDX or Syft) and
   attach them. Consider GitHub artifact attestations for provenance.
7. Leave the release as **draft**. A maintainer promotes it only after clean-VM
   install and update tests. Publishing makes it visible to stable clients.

Use stable tag URLs in generated updater metadata. Passing only a `releaseId`
without `tagName` makes `tauri-action` generate `releases/latest` asset URLs,
which can accidentally point at a later release without updater assets.

Linux Tauri builds require the documented packages, currently including
`libwebkit2gtk-4.1-dev`, `libappindicator3-dev`, `librsvg2-dev`, and `patchelf`.
Do not use `cross` for the desktop shell: WebKit/GTK/AppImage bundling requires
a suitable native environment or a deliberately prepared container/ARM runner.

### Required secrets and environment controls

| Secret | Purpose | Required when |
| --- | --- | --- |
| `TAURI_SIGNING_PRIVATE_KEY` | signs all updater artifacts | every release |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | decrypts updater key | if key is password-protected |
| `APPLE_CERTIFICATE` | base64 Developer ID `.p12` | macOS public release |
| `APPLE_CERTIFICATE_PASSWORD` | imports `.p12` | macOS public release |
| `KEYCHAIN_PASSWORD` | temporary CI keychain | macOS public release |
| `APPLE_ID`, `APPLE_PASSWORD`, `APPLE_TEAM_ID` | notarization via Apple ID | macOS option A |
| `APPLE_API_ISSUER`, `APPLE_API_KEY`, private-key file material | notarization via App Store Connect API | macOS option B (preferred for CI) |
| `WINDOWS_CERTIFICATE`, `WINDOWS_CERTIFICATE_PASSWORD` | PFX signing | Windows option A |
| Azure credentials/account/profile values | cloud-backed signing | Windows option B |

`GITHUB_TOKEN` is provided by Actions and should have only `contents: write` in
the release job. PR jobs need read permissions and must never receive signing
secrets. Put signing secrets in a protected `release` GitHub Environment with
reviewers. Prefer App Store Connect API credentials over an Apple ID password,
and Azure Artifact Signing/Key Vault over exporting a long-lived Windows key
when budget and account availability permit.

## Versioning and changelog

Adopt one SemVer for the desktop app, daemon/CLI, and bundled wire protocol
until compatibility guarantees justify independent packages. A small release
script should update and validate:

- root `Cargo.toml` package version;
- `desktop/src-tauri/tauri.conf.json` version;
- `desktop/src-tauri/Cargo.toml` package version;
- `desktop/package.json` version (and lockfile root package version);
- `CHANGELOG.md` heading and compare links.

Commit those changes as `chore(release): vX.Y.Z`, then create a signed or
protected `vX.Y.Z` tag. Follow Keep a Changelog categories and Conventional
Commits if automated notes are desired. Use `0.x` while APIs and storage are
still moving; never reuse or overwrite a published version/tag because updater
clients cache version decisions and release assets.

Pre-releases (`v0.2.0-beta.1`) should be GitHub prereleases and should not
replace stable `latest.json`. Add a separate beta endpoint/feed only when the
app has an explicit channel preference. GitHub's `/releases/latest` naturally
serves the latest non-prerelease stable release.

## Security, failure, and rollback

- **Trust root:** updater private key is offline/protected; only its public key
  ships in the app. OS signing keys are separate and similarly protected.
- **No partial platform publication:** `latest.json` is validated as a whole by
  Tauri. One incomplete platform entry can break checks for every client.
- **Atomic compatibility:** GUI, daemon, and protocol ship together. Database
  migrations must be transactional and forward-compatible with at least the
  previous app version.
- **Safe restart:** never install while an agent write, merge, push, or service
  transition is active. Preserve recovery state before relaunch.
- **Network failure:** timeout and retry with backoff; the current app remains
  usable. Never delete the installed version before the new artifact verifies.
- **Compromised GitHub account:** signatures still prevent an attacker without
  the updater key from shipping an accepted update. Protect release approval
  and tags anyway.
- **Bad release:** immediately unpublish the bad release so new checks stop
  seeing it, fix forward with a higher patch version, and publish that. The
  default updater rejects lower versions, so replacing `latest.json` with an
  older version is not a rollback.
- **Emergency downgrade:** prefer a manually distributed signed installer. Do
  not globally enable `version_comparator` downgrades; that weakens normal
  monotonic-update guarantees. If an automated rollback is essential, ship the
  capability deliberately in advance with a tightly controlled dynamic feed.
- **Key rotation:** while the old key is still trusted, publish an app version
  that can trust the new key (or implement runtime dual-key rotation), wait for
  adoption, then sign subsequent releases with the new key. Document an offline
  recovery procedure before launch.

## Implementation order and acceptance gates

### Phase 0 — release foundations

1. Decide the first supported platform list and canonical package manager.
2. Unify versions and add `CHANGELOG.md` plus a version validation script.
3. Make the daemon a real same-target Tauri sidecar and prove packaged startup.

Gate: a clean machine can install the bundle, open a project, launch one agent,
quit/reopen the GUI, and reconnect to the daemon.

### Phase 1 — signed manual distribution

1. Enable Tauri bundles.
2. Implement macOS signing/notarization first, then Windows signing.
3. Replace release CI with draft, multi-artifact releases.

Gate: OS signature verification and notarization pass on downloaded artifacts;
CLI installer still works against the same release.

### Phase 2 — auto-update

1. Generate/protect the updater key and configure artifacts/endpoint.
2. Add updater/process plugins, capabilities, and user-facing update flow.
3. Validate `latest.json` and signatures in CI.

Gate: for every supported OS/architecture, install version N, create active and
idle sessions, publish N+1 to a test repository/feed, and verify defer,
download, signature validation, restart, daemon compatibility, and retained
state.

### Phase 3 — production hardening

1. Add SBOM/checksums/attestations and protected release environment.
2. Exercise bad-signature, missing-asset, offline, interrupted-download, and
   failed-daemon-start scenarios.
3. Write and rehearse bad-release, key-loss, and key-compromise runbooks.

Gate: a release owner other than the implementer can execute the checklist and
recover from a deliberately failed update rehearsal.
