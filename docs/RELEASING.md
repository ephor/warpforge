# Releasing Warpforge

Warpforge releases are built from an existing immutable `vX.Y.Z` tag by the
manually triggered **Draft release** GitHub Actions workflow. The workflow
creates a GitHub Release draft, uploads the desktop bundles, updater metadata,
signatures, CLI archives, and checksums, but never publishes the release.

macOS Apple Silicon is the primary release target. Windows x64 and Linux x64
desktop builds are opt-in preview artifacts; a successful CI build does not
mean those platforms have been validated for public support.

## One-time repository setup

Create a protected GitHub Actions environment named `release`. Require a
reviewer for it so signed builds cannot start without approval, and store these
secrets in that environment:

- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` (only when the key has a password)
- `APPLE_CERTIFICATE`
- `APPLE_CERTIFICATE_PASSWORD`
- `APPLE_SIGNING_IDENTITY`
- `APPLE_API_ISSUER`
- `APPLE_API_KEY`
- `APPLE_API_PRIVATE_KEY`

Set the updater public key as the repository Actions variable
`TAURI_UPDATER_PUBLIC_KEY`. It is public by design and is injected into the
Tauri configuration at build time. The corresponding updater private key and
all Apple credentials remain protected environment secrets.

Back up the updater private key and its password in a separate secure vault.
Every installed app trusts that key; losing it prevents those installations
from accepting normally signed future updates. Rotate keys while the old key
is still available by shipping an old-key-signed transition release. If the
private key is lost or compromised before such a transition, recovery may
require users to install a new build manually.

## Local macOS package smoke test

Export the updater private key and password, then bundle both the app and DMG
targets. The public verification key is committed in `tauri.conf.json` because
it is public by design:

```bash
cd desktop
export TAURI_SIGNING_PRIVATE_KEY="$(cat ~/.tauri/warpforge.key)"
read -s "TAURI_SIGNING_PRIVATE_KEY_PASSWORD?Updater key password: "
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD
echo
CI=true bun run tauri -- build -vv \
  --bundles app,dmg \
  --target aarch64-apple-darwin
```

`CI=true` makes Tauri skip Finder-controlled DMG icon positioning. This is
required for reliable local packaging on macOS 26, where Finder may fail to
resolve the temporary mounted volume. It does not affect the application or
updater payload. GitHub Actions already sets `CI=true`.

The updater artifact is generated by the `app` target; building only `dmg`
produces a valid installer but no `.app.tar.gz` updater payload or signature.

## Prepare a version

1. Choose a stable semantic version such as `0.2.0`. Prerelease tags are not
   accepted by the workflow.
2. Set that exact version in all release metadata:
   `Cargo.toml`, `crates/warpforge-protocol/Cargo.toml`,
   `desktop/src-tauri/Cargo.toml`, `desktop/src-tauri/tauri.conf.json`,
   `desktop/package.json`, and regenerate `desktop/bun.lock` when dependencies
   change.
3. Add exactly one `## [X.Y.Z]` heading to `CHANGELOG.md` with user-facing
   release notes.
4. Run the normal CI checks and the release metadata check locally:

   ```bash
   TAURI_UPDATER_PUBLIC_KEY="$(cat ~/.tauri/warpforge.key.pub)" \
     bun scripts/check-release-version.mjs vX.Y.Z
   ```

   Use the same public-key text stored in the repository variable; this command
   never needs the updater private key.

5. Merge the release commit to `main`, then create and push an immutable tag at
   that exact commit:

   ```bash
   git tag -a vX.Y.Z -m "Warpforge vX.Y.Z"
   git push origin vX.Y.Z
   ```

Never move or reuse a release tag. The workflow checks out the tag rather than
the current branch and rejects a checkout or version that does not match it.

## Build the draft

In GitHub Actions, run **Draft release** and enter the existing `vX.Y.Z` tag.
Leave both preview toggles disabled for the normal macOS Apple Silicon release:

- `include_windows_x64` adds Windows x64 NSIS desktop and CLI artifacts.
- `include_linux_x64` adds Linux x64 AppImage, Debian package, and CLI
  artifacts.

The release environment approval gates access to signing credentials. After
approval, the workflow:

- validates the tag, synchronized versions, changelog, updater configuration,
  formatting, tests, lint, and type checks;
- builds the desktop app with its matching daemon bundled as a sidecar;
- signs/notarizes the macOS app and signs every updater artifact;
- builds the CLI archives, including the existing Linux arm64 archive used by
  the shell installer;
- creates or refreshes the exact GitHub Release draft;
- verifies `latest.json`, immutable tagged asset URLs, signatures, expected
  platform entries, and uploads `SHA256SUMS`.

Rerunning the workflow for an unpublished draft removes its old assets first,
so choose the same preview flags intentionally. It refuses to overwrite an
already published release.

## Smoke-test and publish

Do not publish merely because the workflow is green. Download assets from the
draft and test the artifacts that will actually be advertised.

For the primary macOS Apple Silicon release, verify on a clean or representative
machine that:

1. the DMG installs and macOS accepts the signature and notarization;
2. Warpforge launches without a development daemon or build tree;
3. the bundled daemon connects and reports the same application and protocol
   version;
4. a project can be registered, a service can start, and an installed agent can
   launch and exchange a prompt;
5. closing and reopening the desktop app reconnects cleanly.

Packaged Unix launches resolve the user's login/interactive shell environment
and merge its `PATH` with the desktop launcher's inherited entries before
starting the bundled daemon. If startup fails, inspect
`~/.warpforge/logs/desktop-sidecar.log`. It contains only bounded, rotating
sidecar lifecycle and redacted stderr diagnostics; daemon stdout, prompt
content, and authentication tokens are not persisted.

If Windows or Linux previews were requested, smoke-test the same lifecycle on
real machines before describing those platforms as supported. Until that
happens, distribute them explicitly as previews.

Review the generated release notes and asset list, then publish the draft in
GitHub Releases. Publishing makes its `latest.json` available through the
stable updater endpoint. Draft releases are intentionally invisible to that
endpoint, so the real in-app N→N+1 path must be tested immediately after
publication from the previous production version:

1. confirm the update is detected after the daemon connects (or use **Check for
   updates**);
2. download it and verify the displayed version and release notes;
3. with no active work, choose **Restart and update**;
4. confirm the new app relaunches, starts its matching daemon, and reconnects;
5. repeat while an agent task or runtime transition is active and confirm the
   update is refused with blockers rather than interrupting work.

## Updater behavior and safety

The desktop app checks the stable GitHub Releases feed three seconds after its
daemon becomes healthy. It also exposes a manual check. Download and install
are explicit actions; Warpforge never installs an update in the background.

An update contains both the UI and its matching daemon. Before installation,
the app requires an exact app/daemon version and protocol handshake, refuses to
update a daemon started outside the desktop app, blocks new mutations, and asks
the daemon for active-work blockers. Only a clear, desktop-owned daemon is shut
down. The installer runs after that process disconnects, then the app relaunches
and performs the handshake again. If installation fails after daemon shutdown,
the current app is relaunched so it is not left without its runtime.

## Rollback and failed releases

Before publication, fix the source, create a new tag, and discard the bad draft;
do not retag the old commit. After publication, removing or unpublishing a
release can stop additional automatic discovery, but it cannot undo downloads
or installations already completed. The updater is not a downgrade mechanism.
Ship a corrected, higher patch version and keep the previous release assets
available for manual recovery until the replacement is verified.
